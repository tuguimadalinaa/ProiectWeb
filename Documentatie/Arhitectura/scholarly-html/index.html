Abstract

STOL este un API care pune la dispozitia clientilor posibilitatea de gestiona resurse de pe OneDrive, GoogleDrive si DropBox.

Introducere

STOL este una din aplicatiile care vine in ajutorul oricui prin imbinarea mai multor servicii cloud in unul singur.

Scopurile acestei aplicatii web sunt:

  • Folosirea unei singure aplicatii pentru utilizarea mai multor servicii cloud.
  • O buna gestionare a memoriei in toate cele 3 drivere.
  • Fragmentarea si recompunerea fisierelor in functie de spatiul disponibil.

Structura

Tehnologia utilizata

Partea de Backend presupune cod PHP, deoarece este un limbaj usor si care pune la dispozitie multe functionalitati care nu necesita utilizarea suplimentara de Framework-uri.

Stocarea se foloseste de baza de date pusa la dispozitie de XAMPP, MySql.

Suplimentar, sunt utilizare serviciile puse la dispozitie de OneDrive, DropBox si GoogleDrive. Astfel, aplicatia foloseste API-urile acestor servicii pentru gestionarea live a tuturor cerintelor.

Steps:

Crearea unei interfete:

  • Am inceput prin a realiza o interfata cat mai user-friendly. Pentru inceput am facut paginile de start, login si sign up pentru care am implementat in js o simulare de logare.
  • Apoi am ales sa ne ocupam de partea de interfata la nivel de client logat. Aici am creat un menu prin care utilizatorul are access la toate functionalitatile oferite de STOL.
  • Pentru pargina principala avem 3 butoane, fiecare generand un cod de access pentru accesarea de resurse, dar si pt a avea posibilitatea de a face orice operatie pe fisiere.

Crearea de server:

  • Pentru partea de server am folosit arhitectura MVC. Am facut rutare si astfel fiecare request va fi gestionat de catre o ruta diferita pentru o mai buna modularizare.
  • Fiecare request va ajunge intr-o ruta principala de index, care mai apoi va gestiona requestul si il va trimite mai departe.
  • Fiecare request este gestionat de un controller separat, care are cate o functionalitate diferita.

Accesarea drepturilor pt fiecare drive:

  • Pentru fiecare drive am folosit API-urile puse la dispozitie.
  • Primul pas a fost sa gestionam accesul. Astfel fiecare drive trebuie sa obtina un cod de access valid si sa fie transmis mereu in fiecare request.

Gestionarea de fisiere:

  • La nivel de interfata s-a creat cate o pagina pt fiecare drive in care userul isi poate accesa si manageria resursele.
  • La nivel de backend s-a creat un controller pentru fiecare drive. Controllerul acopera toate operatiile de baza care pot fi facute asupra fisierelor.

Crearea de rute speciale pentru API

  • Pentru a putea gestiona accesul am creat rute speciale ce returneaza anumite credentiale (ex. jwt).

Urcarea fragmentata de fisiere:

  • Am pus la dispozitie posibilitatea de a urca fragmentat pe toate driverele un fisier.
  • Astfel daca un drive nu mai are memorie destula pentru un fisier, fisierul se va imparti pe bucati si va fi trimis catre alt drive.
